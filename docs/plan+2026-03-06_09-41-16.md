# 监听主链路优化计划（2026-03-06 09:41:16）

## 目标
- 当前主功能只定义为：监听消息 + 翻译 + 展示。
- 当前主维护链路只包括：
  - `examples/group_listener_worker.py`
  - `examples/sidebar_translate_listener.py`
  - `wechat_auto/window.py`
  - `wechat_auto/controls.py`
- 后续优化默认只服务 `session` 模式。

## 非目标
- 不把发送消息、文件发送、输入框写入作为当前优化目标。
- 不把 `chat` / `mixed` 作为后续性能优化、恢复能力优化、状态监控优化的主路径。
- 不为了兼容旧接口而反向牵制 `session` 主链路设计。

## 当前约束
- `session` 是主路径；`chat` / `mixed` 仅保留兼容。
- 多目标形态保持为：单窗口展示 + 每个 target 一个 worker 子进程。
- 优化必须优先提高稳定性、可恢复性、可观测性，其次才是参数和界面细节。

## 阶段顺序
### 阶段 1：启动等待
- 允许先启动侧边栏程序，再启动微信。
- `group_listener_worker.py` 在微信未启动、未登录、主窗口未就绪时，不直接退出，而是进入等待状态并周期重试。
- `sidebar_translate_listener.py` 需要能稳定展示“等待微信”状态，而不是只显示一次失败文本。

### 阶段 2：窗口恢复
- 微信被关闭后，worker 不能只空转或退出。
- 微信重新打开后，worker 要能重新定位窗口、恢复监听。
- 多 target 下某个 worker 恢复失败时，不能拖垮其他 target。

### 阶段 3：状态监控
- 主进程和 worker 之间的状态要结构化，不再只依赖自由文本日志。
- 至少区分：等待微信、连接中、监听中、窗口丢失、重连中、重连失败退避中。
- 侧边栏状态栏优先展示结构化状态，不用临时日志覆盖掉关键信息。

### 阶段 4：性能优化
- 优先优化 UIA 控件查找，减少每轮全树扫描。
- 对会话列表、消息列表等稳定控件做缓存与失效重建。
- 性能优化不能以增加焦点抢占为代价。

## 设计原则
- 优先保证 `session` 模式稳定，再谈其他模式。
- 恢复逻辑要能覆盖三类情况：
  - 程序先启动，微信后启动
  - 微信运行中被关闭
  - 微信关闭后重新打开
- 日志和状态要能明确区分“未启动”“窗口丢失”“正在重连”“已恢复”。

## 验收标准
- 微信未启动时运行 `examples/sidebar_translate_listener.py`，程序保持存活并持续等待。
- 微信启动并登录后，无需重启程序即可进入监听状态。
- 监听过程中关闭微信，程序不整体退出，状态切换到等待/重连。
- 微信重新打开后，无需重启程序即可恢复监听。
- 多 target 下单个 target 恢复失败，不影响其他 target 的监听与展示。

## 禁止跑偏
- 不把当前迭代时间花在发送链路、输入框、自动回复、旧示例重构上。
- 不为了 `chat` / `mixed` 的边缘行为，把 `session` 主链路重新复杂化。
- 不增加缺少状态定义的“临时补丁式”恢复逻辑。
